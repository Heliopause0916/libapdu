## Project Essence

A single C99 static library, extracted from [OpenSC](https://github.com/OpenSC/OpenSC) `src/libopensc/apdu.c`, retaining only pure APDU parsing and assembly logic.

- Only 2 source files: `include/libapdu/apdu.h` (163 lines) + `src/apdu.c` (309 lines)
- No external dependencies, only C99 standard library required
- No tests, no CI, no lint/format configuration
- License: LGPL-2.1-or-later

## Build

```bash
mkdir build && cd build
cmake ..
cmake --build .
```

- CMake minimum 3.10, static library by default
- Set `-DBUILD_SHARED_LIBS=ON` to build a shared library instead
- Verified to build cleanly with MSVC (toolchain: MSVC 19.50 + Ninja + CMake 4.2)

## Clangd

`.clangd` exists at the project root and is gitignored. To modify clangd configuration, edit `.clangd` directly â€” it won't be tracked by Git.

Key settings: `-std=c99`, `-Iinclude` (header search path), `unused-parameter` diagnostic suppressed.

## API Memory Semantics (Gotchas)

| Function | Caller Responsibility |
|----------|----------------------|
| `apdu_decode()` | After decoding, `apdu->data` points into the input buffer; **caller must keep the input buffer alive while the apdu is in use** |
| `apdu_alloc_and_encode()` | Allocates memory via `malloc()` internally; **caller must `free()`** |
| `apdu_encode()` | Caller provides the output buffer; pre-calculate size via `apdu_get_length()` |

## Struct Notes

- `apdu_t` fields `mac[8]`/`mac_len`, `control`, and `next` pointer are legacy compatibility fields from OpenSC â€” none of the 5 API functions use them
- `apdu_t.resplen`: on input, indicates buffer size; after decoding a response, updated to the actual returned data length

## No Tests & No CI

The project currently has zero test code or CI configuration. If adding them:
- Testing: consider CMake's built-in `enable_testing()` + simple C test files
- CI: GitHub Actions (Linux + Windows matrix build)

## .clangd vs compile_commands.json

- `compile_commands.json` (generated by CMake) is gitignored
- `.clangd` manually adds `-Iinclude` and `-std=c99` to compensate for the missing compilation database
- If you re-run CMake and generate `compile_commands.json`, clangd will automatically prefer it over `.clangd`

## Windows/MSVC Notes

- The `Remove: [-m*]` in `.clangd` filters GCC/Clang-specific flags like `-march`/`-mtune`, preventing false diagnostics under MSVC
- The project compiles successfully under MSVC with no special compatibility workarounds
